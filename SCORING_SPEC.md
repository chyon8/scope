# CaseLab Completion 점수 산정 기준표 v2

> **최종 목표:** 검수매니저가 상담하며 메모/녹취를 남기면, AI가 분석하여 점수를 산출하고, 부족한 정보에 대한 질문을 생성한다. 점수가 충족되면 상담을 종료하고, **개발사가 추가 미팅 없이 견적을 제안할 수 있는 수준의 공고문(draft)**을 출력한다.

---

## 1. 최종 산출물: 공고 양식 (Output Format)

주신 예시 데이터 7개에서 추출한 실제 공고 양식. **이것이 최종 목표물이고, 점수 기준은 이 양식이 채워지는지를 역산한 것.**

| # | 공고 섹션 | 필수 여부 |
|---|----------|----------|
| 1 | **프로젝트 개요** | 필수 |
| 2 | **프로젝트 배경 및 목표** | 필수 |
| 3 | **과업 범위** — 수행 범위 | 필수 |
| 4 | **과업 범위** — 상세 기능 요구사항 (모듈별) | 필수 |
| 5 | **과업 범위** — 관리자 기능 | 필수 |
| 6 | **비기능 요구사항** (성능/보안) | 선택 |
| 7 | **기술/제조 스택** | 필수 |
| 8 | **클라이언트 준비 사항** | 선택 |
| 9 | **주요 일정** | 선택 |
| 10 | **지원 자격 및 우대 사항** | 필수 |
| 11 | **산출물** | 필수 |
| 12 | **계약 관련 특이사항** | 선택 |

---

## 2. 내부 분석 섹션 (Scoring Layer)

공고의 각 섹션을 채우기 위해 **내부적으로 더 세밀하게 분석**하는 단위. AI는 이 12개 섹션 각각의 confidence(0~100)를 평가한다.

### 섹션 정의, 가중치, 공고 매핑

| # | 섹션 ID | 내부 분석 항목 | 가중치 | → 공고 양식 매핑 |
|---|---------|--------------|--------|-----------------|
| 1 | `purpose` | 프로젝트 목적/개요 | **15%** | 프로젝트 개요, 프로젝트 배경 및 목표 |
| 2 | `core_problem` | 핵심 문제/현재 운영방식 | **10%** | 프로젝트 배경 및 목표 |
| 3 | `features` | 사용자 핵심 기능 | **20%** | 과업 범위 → 상세 기능 요구사항 |
| 4 | `admin` | 관리자 기능 | **10%** | 과업 범위 → 관리자 기능 |
| 5 | `users` | 타겟 사용자/규모 | **5%** | 프로젝트 개요, 지원 자격 |
| 6 | `platform` | 플랫폼/개발 범위 | **10%** | 과업 범위 → 수행 범위 |
| 7 | `integrations` | 외부 연동 | **5%** | 과업 범위 → 상세 기능, 기술 스택 |
| 8 | `design` | 디자인 범위 | **5%** | 과업 범위 → 수행 범위, 클라이언트 준비 사항 |
| 9 | `tech_stack` | 기술 스택/인프라 | **5%** | 기술/제조 스택 |
| 10 | `budget` | 예산 | **5%** | 계약 관련 특이사항 |
| 11 | `timeline` | 일정 | **5%** | 주요 일정 |
| 12 | `deliverables` | 산출물/자격요건/우대사항 | **5%** | 산출물, 지원 자격 및 우대 사항 |

**합계: 100%**

### 매핑 역방향 (공고 섹션 → 어떤 내부 섹션들이 채워져야 하는가)

| 공고 섹션 | 채워지려면 필요한 내부 섹션 | 비고 |
|----------|------------------------|------|
| 프로젝트 개요 | `purpose`, `users` | |
| 프로젝트 배경 및 목표 | `purpose`, `core_problem` | |
| 과업 범위 — 수행 범위 | `platform`, `design` | 웹/앱, 신규/고도화, 디자인 포함 여부 |
| 과업 범위 — 상세 기능 | `features`, `integrations` | 모듈별 기능 명세 |
| 과업 범위 — 관리자 기능 | `admin` | |
| 비기능 요구사항 | `features`에서 파생 | 성능/보안은 AI가 프로젝트 성격에서 추론 |
| 기술/제조 스택 | `tech_stack` | |
| 클라이언트 준비 사항 | `design`, `integrations` | Figma 유무, API 문서 유무 등 |
| 주요 일정 | `timeline` | |
| 지원 자격 및 우대 사항 | `deliverables`, `integrations`, `platform` | 필요 역량은 기능/연동에서 파생 |
| 산출물 | `deliverables` | |
| 계약 관련 특이사항 | `budget`, 기타 | |

> [!IMPORTANT]
> **이 매핑이 핵심.** 내부 섹션 `features`의 confidence가 올라가면, 공고의 "상세 기능 요구사항"뿐 아니라 "비기능 요구사항"과 "지원 자격"도 함께 작성 가능해진다. 1개의 내부 섹션이 여러 공고 섹션에 기여하고, 1개의 공고 섹션이 여러 내부 섹션을 필요로 한다.

## 3. 섹션별 Confidence 평가 기준

### `purpose` (프로젝트 목적/개요) — 15%

| 점수 | 기준 | 공고에 미치는 영향 |
|------|------|-----------------|
| 0~20 | "앱 만들어주세요" 수준 | 개요, 배경 모두 작성 불가 |
| 21~50 | 대략적 성격 파악 가능하나 비즈니스 배경 불명확 | 개요만 한 줄 작성 가능 |
| 51~80 | 무엇을 + 왜 만드는지 서술됨 | 개요 + 배경 초안 작성 가능 |
| 81~100 | 비즈니스 목표, 기대 효과, 개선점이 구체적 | 개요 + 배경 완성 가능 |

**예시 검증:**
- 데이터 1 (아이허브) 완성본: "제휴 수익 모델 다각화 + 해외 사용자 유입 + 로컬라이징 인프라 마련" → **90점**
- 데이터 2 (TM CRM) 초기: "앱이 필요합니다" → **25점**
- 데이터 3 (프로토타입): "정부지원사업 검증용 데이터 수집이 목적" → **85점**

---

### `core_problem` (핵심 문제/현재 운영방식) — 10%

| 점수 | 기준 | 공고에 미치는 영향 |
|------|------|-----------------|
| 0~20 | 현재 운영방식 언급 없음 | "배경 및 목표" 절반 비어있음 |
| 21~50 | "기존 시스템 없음" 정도 확인 | 배경 한 줄 작성 가능 |
| 51~80 | 운영 방식 + 불편한 점 서술 | 배경 본문 작성 가능 |
| 81~100 | 구체적 페인포인트 + 디지털화 필요 이유 명시 | 배경 및 목표 풍부하게 작성 |

**예시 검증:**
- 데이터 2 (TM CRM) 완성본: "개인 폰 + 수기 엑셀 → 정확성/효율성 저하, 고연령 마케터라 직관적 UI 필수" → **95점**
- 데이터 4 (요양병원): "수기 발주 + 병원마다 상이한 단가 + ERP 엑셀 수동 처리" → **95점**

---

### `features` (사용자 핵심 기능) — 20% ⭐ 최대 가중치

| 점수 | 기준 | 공고에 미치는 영향 |
|------|------|-----------------|
| 0~20 | 기능 언급 거의 없음 | 상세 기능 요구사항 작성 불가 |
| 21~40 | 핵심 기능 1~2개 추상적 언급 | 기능 나열만 가능, 모듈 분리 불가 |
| 41~60 | 주요 기능 3~5개 나열, 상세 흐름 없음 | 기능 목록은 작성 가능, 모듈별 상세 불가 |
| 61~80 | 모듈별 구분 + 사용자 흐름 대략 파악 | 모듈별 상세 기능 초안 작성 가능 |
| 81~100 | 모듈별 구조화 + 입출력/동작 방식 구체적 | 개발사가 바로 이해 가능한 상세 기능 명세 |

**예시 검증:**
- 데이터 1 완성본: 쿠폰 연동 모듈(API 수신, 프로모션 UI, 보관 시스템, 상태 변경, 발급 제한) + 영문 전환 모듈 → **90점**
- 데이터 4 완성본: 회원관리, B2B 발주, 관리자 대시보드, ERP 엑셀 연동 — 4개 모듈, 각각 5~6개 세부 기능 → **95점**
- TM CRM 초기: "고객 관리, 영업 관리, 통계 등" 나열 → **30점**

## 4. Completion 계산

### 공식

```
overall_completion = Σ (section_confidence × section_weight)
```

**서버에서 계산. AI에게 맡기지 않음.**

```javascript
const SECTION_WEIGHTS = {
  purpose: 0.15,
  core_problem: 0.10,
  features: 0.20,
  admin: 0.10,
  users: 0.05,
  platform: 0.10,
  integrations: 0.05,
  design: 0.05,
  tech_stack: 0.05,
  budget: 0.05,
  timeline: 0.05,
  deliverables: 0.05
};

function calculateCompletion(sections) {
  let total = 0;
  for (const [key, weight] of Object.entries(SECTION_WEIGHTS)) {
    const confidence = sections[key]?.confidence || 0;
    total += confidence * weight;
  }
  return Math.round(total);
}
```

## 5. 상담 종료 판별

### 등급

| 등급 | 점수 | 의미 |
|------|------|------|
| 🔴 초기 | 0 ~ 39 | 핵심 정보 대부분 미확보 |
| 🟡 진행 중 | 40 ~ 69 | 뼈대는 잡혔으나 견적 내기엔 부족 |
| 🟢 등록 가능 | 70 ~ 84 | 공고 등록 가능, 일부 보완 권장 |
| 🔵 완성 | 85 ~ 100 | 추가 미팅 없이 견적 제안 가능 |

### 상담 종료 조건

```
상담 종료 가능 = overall_completion ≥ 70
              AND features.confidence ≥ 50
              AND purpose.confidence ≥ 50
              AND platform.confidence ≥ 50
```

> [!IMPORTANT]
> 전체 점수가 70이어도 핵심 3개(목적/기능/플랫폼) 중 하나라도 50 미만이면 종료 불가. "예산/일정만 잘 적었는데 기능은 안 적은" 기형적 공고 방지.

## 6. 질문 생성 우선순위

### 공식

```
질문 우선도 = section_weight × (100 - section_confidence)
```

가중치 높은데 confidence 낮은 섹션 = 가장 먼저 질문.

### 규칙

1. **한 번에 최대 3개 질문** — 적은 질문으로 많은 걸 끌어냄
2. **우선도 상위 3개 섹션에 대해서만 생성**
3. **confidence ≥ 80인 섹션은 질문하지 않음**
4. **하나의 질문으로 여러 섹션을 커버 가능하면 그렇게**
5. **매니저가 그대로 읽어도 자연스러운 말투**

### 예시 계산

| 섹션 | 가중치 | confidence | 우선도 |
|------|--------|-----------|--------|
| features | 0.20 | 10 | **18.0** ← 1순위 |
| purpose | 0.15 | 30 | **10.5** ← 2순위 |
| admin | 0.10 | 0 | **10.0** ← 3순위 |
| platform | 0.10 | 50 | 5.0 |
| budget | 0.05 | 0 | 5.0 |

→ AI는 features → purpose → admin 순서로 질문 생성

## 7. UI 표시 방식: 공고 양식 기준으로 묶어서 표시

매니저 화면에서는 내부 12개 섹션이 아닌, **공고 양식 단위로 묶어서** 보여줌.

### 공고 섹션별 완성도 (매니저에게 보이는 화면)

```
📋 공고 섹션별 현황

프로젝트 개요           ████████░░ 80%  ✅
  └ 기반: purpose(90), users(60)

프로젝트 배경 및 목표    ████████░░ 82%  ✅
  └ 기반: purpose(90), core_problem(75)

과업 범위 — 수행 범위    ███████░░░ 65%  ⚠️
  └ 기반: platform(80), design(20)    ← 디자인 범위 미확인

과업 범위 — 상세 기능    ██████░░░░ 55%  ⚠️
  └ 기반: features(60), integrations(30)
```

## 10. 전체 흐름 정리

```
고객 초기 내용 등록
  ↓
검수매니저에게 배정
  ↓
매니저가 고객과 상담 (전화/채팅)
  ↓
메모 입력 (향후 STT 녹취)
  ↓
AI 분석 → 12개 섹션 confidence 산출
  ↓
서버: 가중 합산 → completion 계산
서버: 공고 양식 매핑 → UI 표시용 점수 계산
서버: 상담 종료 가능 여부 판별
  ↓
매니저 화면:
  - 공고 양식 기준 섹션별 완성도 게이지
  - 추천 질문 3개
  - 상담 종료 버튼 (활성/비활성)
  ↓
매니저가 추천 질문으로 추가 상담 → 메모 → AI 재분석 (반복)
  ↓
completion ≥ 70 + 핵심 섹션 충족
  ↓
[상담 종료] → AI가 공고 양식에 맞춰 최종 draft 생성
  ↓
공고 등록
```
