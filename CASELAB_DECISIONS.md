# CaseLab 개발 논의 및 결정 사항

이 문서는 대화를 통해 확정된 CaseLab 프로젝트의 주요 방향성과 의사결정 내용을 잊지 않기 위해 기록한 것입니다.

## 1. 데이터 소스 및 권한 연동
- **녹취/통화 API**: 연동을 위한 API 주소와 인증 키만 필요합니다.
- **본진 DB (Read Replica)**: 호스트(IP/Domain), 포트(Port), 데이터베이스 이름, 계정명(Username), 비밀번호(Password)가 필요합니다. 내부망에 위치한 경우 VPN 접속 권한이나 작업 환경의 IP 화이트리스트 추가가 요구될 수 있습니다.

## 2. 작업 우선순위 및 대기(Hold) 사항
- **녹취 데이터 연동**: 우선 통화 API를 통해 기존에 추출된 텍스트(STT) 데이터와 요약을 받아와 화면에 띄우는 **파이프라인 구축**까지만 진행합니다.
- **프롬프트 작업 대기**: 리스크, 범위 변경 등 핵심 요소를 추출하기 위한 AI 프롬프트 고도화 작업은 사용자가 프롬프트를 직접 확인하고 검토할 때까지 **진행하지 않고 대기**합니다.

## 3. 유사 사례 검색용 데이터 범위
- 12만 건의 전체 데이터를 한 번에 넣지 않고, **최근 1년 치** 데이터로 범위를 좁혀서 진행하기로 결정했습니다.
- 이를 통해 초기에는 벡터 DB와 같은 무거운 인프라 구축을 미루고 가볍게 검증할 수 있습니다.

## 4. UI/디자인 대폭 변경 시 프로젝트 관리 방식
- 새로운 프로젝트(레포지토리)를 생성하는 대신, **현재 프로젝트 기반 위에서 덮어쓰거나 브랜치를 분리하여 이어가는 방식**을 채택합니다.
- 기존 기반(데이터 모델, 백엔드 API 등)이 같기 때문에 재사용성을 높이기 위함입니다.
- **추천 액션**: Git 브랜치(예: `feat/new-ui`)를 새로 파거나, 컴포넌트 폴더를 분리하여 기존 코드를 참고하면서 새 컴포넌트로 점진적으로 교체합니다.

## 5. 실데이터 연동 및 상태 동기화 방식
- **데이터 연동 (Read)**: DB 계정을 소스코드에 직접 넣는 대신, 과거에 구축해 둔 **n8n 웹훅(API)**을 호출하여 DB 데이터를 가져오는 방식을 우선적으로 고려합니다.
- **어댑터 패턴 (개발 방식)**: 실데이터 주소(n8n 등)가 확정되기 전까지는 가짜 주소(`mock`)를 바라보며 개발을 진행합니다. 향후 실주소가 준비되면 어댑터 스위치만 갈아끼워 완벽하게 연동되도록 설계합니다.
- **상태 동기화 (Sync)**: '검수중' -> '모집중' 등 본진 DB의 상태 변경을 CaseLab에 반영하기 위해, 추후 n8n을 주기적으로 조회(Polling)하거나 n8n에서 변경 이벤트를 쏴주는(Webhook Push) 방식을 사용합니다.
